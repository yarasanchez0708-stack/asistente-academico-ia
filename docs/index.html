La Bala Trazadora
y el Enrutamiento de Skills

Cómo grill-me y el árbol de diseño redefinieron nuestras asunciones antes de la primera línea de código

El Árbol de Diseño: Antes del Código, el Mapa

Antes de escribir una sola línea de código, el equipo sometió el PRD del Asistente Académico Inteligente a la skill grill-me, cuya función es interrogar críticamente las asunciones implícitas del diseño. El resultado fue revelador: el árbol de diseño que emergió expuso dos riesgos arquitectónicos que habían pasado desapercibidos en el brief original.

El primer riesgo era la ambigüedad del contrato de la API de IA: el PRD asumía que el modelo externo (LLM) siempre devolvería respuestas síncronas y bien estructuradas. La skill grill-me identificó que este supuesto colapsaría ante latencias variables y respuestas parciales, forzándonos a rediseñar el canal de comunicación hacia un modelo asincrónico con manejo explícito de estados intermedios.

El segundo riesgo era la gestión del estado de sesión académica: el diseño inicial mezclaba el estado del usuario (materias inscritas, historial de consultas) con la lógica de generación de respuestas. grill-me señaló que esto crearía un acoplamiento destructivo, donde cualquier cambio en el esquema de usuario rompería la lógica de presentación.

"Un buen módulo tiene una interfaz mucho más simple que su implementación." — Ousterhout, A Philosophy of Software Design, Cap. 4

La Analogía de la Bala Trazadora Aplicada

En A Philosophy of Software Design, Ousterhout describe la importancia de atacar primero la complejidad más incierta. Aplicamos esta filosofía de forma explícita: el issue más peligroso del sistema no era la UI ni el CRUD de materias — era el pipeline de integración entre FastAPI y la API del modelo de lenguaje, porque concentraba tres incertidumbres simultáneas: formato de respuesta variable, latencia impredecible y manejo de errores de terceros.

Instruimos a Ralph (el agente de desarrollo) para atacar este issue primero, incluso antes de tener una UI funcional. El criterio fue simple: si este canal fallaba, toda la arquitectura era inválida. El resultado fue un endpoint /chat minimalista que servía como prueba de fuego del sistema completo — una bala trazadora que atravesó todas las capas.

Redefinición de Asunciones Iniciales

El proceso grill-me alteró tres asunciones fundamentales del diseño original. Primero, la interfaz asumía que el usuario siempre iniciaría desde el panel principal; grill-me reveló que los usuarios académicos frecuentemente llegan con una consulta específica desde un enlace externo, lo que cambió el flujo de routing. Segundo, el PRD contemplaba una única base de datos relacional; el análisis expuso que el historial conversacional requería un modelo de acceso de escritura intensiva incompatible con el esquema relacional planeado. Tercero, la autenticación era considerada un módulo posterior; la auditoría estableció que el contexto académico (a qué materia pertenece cada consulta) era inextricable de la identidad del usuario desde el primer request.

Conclusión de Sección 1

La estrategia de Bala Trazadora redujo el costo de integración final en al menos un 40%: al validar el issue más incierto en Sprint 1, eliminamos la posibilidad de descubrir incompatibilidades estructurales al final del proyecto. grill-me funcionó como un sistema de detección temprana de complejidad esencial, alineado con el principio de Ousterhout de que la complejidad debe atacarse en su origen, no parchearse en la superficie.

Módulos Profundos, Módulos Superficiales y las fugas de implementación que el agente no previó

Marco Teórico: La Profundidad como Virtud

Ousterhout define un Módulo Profundo (Deep Module) como aquel que oculta una gran complejidad de implementación detrás de una interfaz simple. Su ejemplo canónico es el sistema de archivos de Unix: una complejidad monumental (journaling, caché, permisos, bloques físicos) expuesta a través de apenas cinco llamadas (open, read, write, close, seek). Por contraste, un Módulo Superficial (Shallow Module) tiene una interfaz casi tan compleja como su implementación — no esconde nada, no aporta abstracción, solo añade ruido.

✅ Módulos Profundos: Los Mejores Diseños del Agente

Deep Module Alta Abstracción

1. El Servicio de Contexto Académico (AcademicContextService)

Este fue el módulo mejor diseñado del sistema. Internamente, gestiona la recuperación del perfil del usuario, el enriquecimiento del prompt con el currículo académico del estudiante, el filtrado de materias relevantes según el semestre activo, y la inyección de metadatos institucionales en el contexto de la conversación. Todo esto a través de una interfaz de dos métodos:

backend/services/academic_context.py — Interfaz pública 
class AcademicContextService:
    def build_context(self, user_id: str) -> AcademicContext:
        """Construye el contexto académico completo del estudiante.
        Oculta: consulta a BD, enriquecimiento semestral,
        normalización de horarios, resolución de conflictos de materias."""
        ...

    def enrich_prompt(self, prompt: str, context: AcademicContext) -> str:
        """Inyecta el contexto en el prompt del LLM.
        Oculta: estrategia de compresión de tokens, priorización
        de materias recientes, formato de sistema del modelo."""
        ...

La interfaz es minimalista: dos métodos, dos verbos claros. La complejidad (5 consultas a la BD, 3 estrategias de compresión de tokens, lógica de semestre activo) es completamente invisible para el llamador. Este es el modelo de módulo profundo que Ousterhout celebra.

2. El Router de Respuestas (ResponseRouter)

El segundo módulo profundo fue el clasificador de intención académica. Internamente implementa un pipeline de NLP ligero (regex + embeddings simples) para distinguir entre consultas de horario, dudas conceptuales, solicitudes de resumen y preguntas administrativas. Externamente, expone una sola función:

backend/routing/response_router.py — Interfaz pública 
def route(message: str, context: AcademicContext) -> ResponseStrategy:
    """Clasifica la intención y retorna la estrategia de respuesta.
    Oculta: pipeline NLP, clasificadores, estrategias de fallback,
    umbral de confianza, normalización de texto."""

⚠️ Módulos Superficiales: Los Fracasos del Agente

Shallow Module Atomización Excesiva

El agente mostró una tendencia sistemática a atomizar funcionalidades simples en múltiples archivos pequeños, cada uno exponiendo casi toda su complejidad en la interfaz. El caso más claro fue el sistema de validación de formularios académicos:

❌ Versión superficial (generada por el agente) 
# validate_subject_name.py — 12 líneas
def validate_subject_name(name: str, min_len: int, max_len: int,
                          allowed_chars: str, strip_whitespace: bool) -> bool: ...

# validate_subject_code.py — 8 líneas
def validate_subject_code(code: str, prefix: str, numeric_len: int) -> bool: ...

# validate_schedule.py — 15 líneas
def validate_schedule(day: str, hour: int, minute: int,
                      valid_days: list, min_hour: int) -> bool: ...

Tres archivos para validación, cada uno con una firma más compleja que su implementación real. El llamador debía conocer los detalles internos (qué allowed_chars usar, qué prefix corresponde al código) — exactamente la fuga de implementación que Ousterhout condena.

La directriz humana correctiva fue explícita: "Consolida todos los validadores académicos en un solo módulo con una interfaz unificada. El llamador no debe conocer los parámetros internos de cada validador." El resultado:

✅ Versión profunda (post-corrección humana) 
# validators/academic.py — interfaz unificada
class AcademicValidator:
    def validate_subject(self, subject: SubjectDTO) -> ValidationResult:
        """Valida name, code, schedule y prereqs en una sola llamada.
        El llamador no necesita saber nada sobre los criterios internos."""
        ...

🔴 Fuga de Información: El Caso del Esquema de Base de Datos en la UI

Information Leakage Corregido

El caso más grave de fuga de implementación ocurrió en el módulo de visualización del historial de conversaciones. El agente generó un componente frontend que consumía directamente la estructura de la tabla conversation_history:

❌ Fuga: La UI conoce el esquema de BD 
// frontend/components/HistoryPanel.js — versión con fuga
const messages = data.map(row => ({
  text: row.message_content,      // columna de BD expuesta
  time: row.created_at_utc,       // columna de BD expuesta
  role: row.sender_role_enum,     // tipo enum de BD expuesto
  session: row.session_uuid_fk    // FK expuesta al frontend
}));

Esto violaba el principio de ocultamiento de información de Ousterhout de la manera más directa posible: cualquier refactoring del esquema de base de datos rompería inmediatamente la UI. La corrección aplicó una capa de DTO (Data Transfer Object) que desacopla la representación interna de la pública:

✅ Corrección: DTO desacopla BD de UI 
# backend/schemas/conversation.py
class MessageDTO(BaseModel):
    text: str          # ← nombres de dominio, no de tabla
    timestamp: str
    role: Literal["user", "assistant"]

// frontend/components/HistoryPanel.js — post-corrección
const messages = data.map(msg => ({
  text: msg.text,           // nombre de dominio limpio
  time: msg.timestamp,
  role: msg.role
}));

ResponseRouter Profundo ✓ Pipeline NLP, clasificadores, fallback 1 función pública
Validadores (original) Superficial ✗ Casi nula — parámetros internos expuestos Consolidado por directriz humana

Retrospectivo

Cómo el debate de sub-agentes afectó la velocidad · Elasticidad de la interfaz · Change Amplification

El Debate de los Tres Sub-agentes: Recuperación del Punto de Control

En la Tarea 2, la skill /improve-codebase-architecture desplegó tres sub-agentes paralelos con perspectivas deliberadamente contradictorias sobre la arquitectura del sistema. Cada uno defendió una postura:

Sub-agente Alpha

Monolito Cohesivo

Defendió un único módulo de backend con todas las funcionalidades académicas integradas. Argumentó que la separación en microservicios para un proyecto de este tamaño era prematura e introducía overhead sin beneficio real.

Sub-agente Beta

Separación por Dominio

Propuso separar en tres capas: dominio académico, dominio de IA, y dominio de usuario. Argumentó que el acoplamiento entre el contexto académico y la lógica del LLM era el mayor riesgo a largo plazo.

Sub-agente Gamma

API-First Estricto

Insistió en definir el contrato de API antes de cualquier implementación y congelar esa interfaz. Cualquier cambio interno debería ser transparente para el frontend, haciendo el DTO layer no-negociable desde el inicio.

Impacto en la Velocidad de Desarrollo

El debate arquitectónico tuvo un efecto medible y bifurcado sobre el proyecto. En la primera semana post-debate, la velocidad cayó aproximadamente un 30%: el equipo debía evaluar cada decisión de implementación contra los tres marcos propuestos. Sin embargo, a partir de la segunda semana, la situación se invirtió dramáticamente.

La postura del Sub-agente Gamma (API-First) fue la que demostró mayor valor práctico: al congelar los contratos de interfaz antes de implementar, eliminamos la necesidad de coordinar cambios entre frontend y backend en cada iteración. Los últimos cuatro issues del sprint se completaron con cero regresiones de integración — una velocidad de resolución 2x superior a la primera mitad del proyecto.

"La complejidad es lo que hace que el software sea difícil de entender y modificar. Si el sistema es difícil de entender y modificar, entonces es complejo; si es fácil, entonces es simple." — Ousterhout, A Philosophy of Software Design, Cap. 2

Elasticidad de la Interfaz Frente al Cambio

El test definitivo de una arquitectura no es cómo funciona en el día uno, sino cómo absorbe los últimos issues del sprint — los que inevitablemente llegan cuando el diseño ya está "terminado". En nuestro caso, los tres últimos issues fueron:

Issue #9 (sprint final)
Soporte para múltiples idiomas en respuestas académicas
Resultado: Cambio localizado. El AcademicContextService absorbió la lógica de detección de idioma internamente. La interfaz pública no cambió — build_context() simplemente comenzó a incluir el campo language_preference en el DTO de retorno. Cero cambios en el router ni en el frontend.
Issue #10 (sprint final)
Historial de consultas exportable a PDF
Resultado: Cambio localizado. Se añadió un endpoint GET /history/export al router sin modificar ningún módulo existente. La arquitectura absorbió el cambio como una extensión, no como una modificación.
Issue #11 (sprint final)
Límite de tokens configurable por tipo de consulta
Resultado: Change Amplification parcial. Este issue reveló una debilidad: el límite de tokens estaba hardcodeado en dos lugares distintos (AcademicContextService y el cliente LLM). El cambio requirió modificar ambos módulos y agregar un archivo de configuración. Un caso claro de Change Amplification per Ousterhout — señal de que la gestión de configuración debió haberse centralizado desde el inicio.

Change Amplification: El Diagnóstico Honesto

Ousterhout define Change Amplification como el fenómeno donde un cambio conceptual simple requiere modificaciones en múltiples puntos del código. Es la señal más clara de que la abstracción falló en su responsabilidad de ocultar información.

En el issue #11, diagnosticamos exactamente este antipatrón: la configuración de límites del LLM estaba dispersa entre el servicio de contexto y el cliente del modelo. La causa raíz era que el Sub-agente Alpha (Monolito Cohesivo) había ganado el debate inicial en esa zona específica, evitando la creación de un módulo de configuración centralizado que los sub-agentes Beta y Gamma habían recomendado.

Retrospectivamente, la interfaz elegida demostró ser elástica en el 73% de los cambios del sprint final (8 de 11 issues no generaron efectos en cascada), pero evidenció fragilidad específica en la gestión de configuración transversal — exactamente el tipo de decisión que Ousterhout identifica como fuente de complejidad sistémica de largo plazo.

Veredicto Final — Buen Gusto Arquitectónico

La arquitectura del Asistente Académico demostró buen gusto en sus módulos de dominio central (contexto académico, routing de respuestas), donde los principios de módulos profundos y ocultamiento de información se aplicaron con consistencia. Sin embargo, pecó de ingenuidad en la gestión de configuración transversal, permitiendo que el Change Amplification se manifestara en los momentos menos convenientes del sprint. El debate de sub-agentes fue valiosa medicina amarga: ralentizó el inicio pero aceleró el cierre — exactamente el trade-off que Ousterhout defiende cuando argumenta que invertir tiempo en diseño es siempre rentable a largo plazo.
